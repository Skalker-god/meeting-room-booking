# Система бронювання переговорної кімнати — Hexagonal Architecture

## 1. Назва роботи

Система бронювання переговорної кімнати з перевіркою перетину часових слотів.

## 2. Опис use case

Система дозволяє користувачу забронювати переговорну кімнату на певний часовий проміжок.

Основний сценарій роботи:

1. Користувач вводить ідентифікатор кімнати.
2. Користувач вводить час початку та завершення бронювання.
3. Система перевіряє, чи не перетинається нове бронювання з уже існуючими бронюваннями для цієї кімнати.
4. Якщо перетину немає — бронювання створюється.
5. Якщо часові інтервали перетинаються — система повідомляє про помилку.

Таким чином реалізується бізнес-правило:
**неможливо створити бронювання, якщо часовий слот уже зайнятий.**

---

## 3. Обрана архітектура

У проєкті використано **Hexagonal Architecture (Ports and Adapters)**.

Цей підхід передбачає розділення системи на внутрішню бізнес-логіку та зовнішні компоненти, які взаємодіють із нею через інтерфейси (порти).

---

## 4. Обґрунтування вибору архітектури

Hexagonal Architecture дозволяє:

* чітко відокремити бізнес-логіку від технічних деталей;
* легко змінювати інфраструктурні компоненти (наприклад, базу даних);
* забезпечити зрозумілу структуру системи;
* зробити систему більш модульною та тестованою.

Для цієї лабораторної роботи ця архітектура підходить, оскільки дозволяє продемонструвати розподіл відповідальностей між компонентами системи.

## 6. Розподіл відповідальностей

### Domain layer

Містить основну бізнес-логіку системи.

Файл:

```
domain/booking.py
```

Тут описана сутність **Booking**, яка представляє бронювання переговорної кімнати.
Також міститься метод перевірки перетину часових слотів.

---

### Application layer

Файл:

```
application/book_room_use_case.py
```

Тут реалізовано основний **use case — бронювання кімнати**.

Use case:

* створює нове бронювання;
* отримує список існуючих бронювань;
* перевіряє перетин часових інтервалів;
* зберігає бронювання, якщо перевірка пройдена.

---

### Ports (інтерфейси)

Файл:

```
ports/booking_repository.py
```

Це інтерфейс, через який бізнес-логіка працює зі сховищем даних.

Use case не знає, як саме зберігаються дані.

---

### Adapters (інфраструктура)

Файл:

```
adapters/in_memory_booking_repository.py
```

Це реалізація порту `BookingRepository`.

У цьому проєкті використано просте сховище **в пам'яті (in-memory storage)**.

---

### Interface layer

Файл:

```
interface/console_controller.py
```

Це шар взаємодії з користувачем.

Він:

* отримує дані від користувача через консоль;
* викликає use case;
* виводить результат виконання.

---

## 7. Бізнес-правила системи

У системі реалізовано наступне бізнес-правило:

**Новий часовий слот бронювання не повинен перетинатися з уже існуючими бронюваннями для тієї ж кімнати.**

Якщо перетин виявлено, система не дозволяє створити бронювання.

---

## 8. Як запустити проєкт

1. Встановити Python.

2. Клонувати репозиторій:

```
git clone <посилання_на_репозиторій>
```

3. Перейти у папку проєкту:

```
cd meeting-room-booking
```

4. Запустити програму:

```
python main.py
```

5. Ввести дані для бронювання через консоль.
